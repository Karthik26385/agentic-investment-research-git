# Workflow: Historical Backtesting Engine (v1.0)

---

## Scope & Limitations — READ BEFORE RUNNING

**This is v1.0 — a plumbing and logic stress-test. It is NOT a statistically valid backtest.**

### Look-Ahead Bias (Unfixable in v1.0)

Claude's training data contains market outcomes from after any historical date being simulated. Even with the information-cutoff rule in Step 3.1, Claude's reasoning is contaminated by embedded knowledge of subsequent events. Any win rate, EV, or return figure produced by this engine cannot be used as evidence of a real statistical edge.

### What This Validates
- CT gate (counter_thesis.md) correctly filters bad trade ideas on historical data
- Sector exposure caps (40%/50%) are enforced during the simulation
- R:R thresholds reject low-quality setups as intended
- Outcome tracking logic (stop / target hit detection) works correctly
- Output file naming and session ID threading works end-to-end

### What This Does NOT Validate
- Signal quality or predictive accuracy
- Win rate or expected value (look-ahead bias taints all numbers)
- Whether this system would have made money historically

### Integration Boundaries
- Writes to: `state/trade_history.json` (append, tagged `BACKTEST`) and `state/signal_performance.json` (append, tagged `BACKTEST`)
- Never modifies: `state/portfolio_ledger.json`, `state/regime_state.json` weights, `state/sector_exposure.json` live exposure, `state/thesis_queue.json`, or any `workflows/*.md` file
- Backtest days do NOT count toward the ≥10 simulation days required for EXECUTION mode

### v2.0 Note
v2.0 will be a Python rebuild with deterministic rule-based signals, real OHLC data (yfinance / Polygon.io / Alpaca), Claude used only for qualitative CT gate review, reproducible results, and transaction cost modeling. The current file is a bridge until that rebuild is complete.

---

## Objective

Simulate past N trading days to stress-test the system's filters (CT gate, sector caps, R:R rules), validate outcome tracking logic, and produce a structured report for human review — without making automated changes to any live system state.

---

## When to Run

Triggered by the command `Start BACKTESTING N Days` or `Start BACKTESTING` (default N = 10).

The orchestrator (Step 0) handles mode switching, SESSION_ID generation, and routing. This file executes when `active_mode == BACKTESTING`.

```
Required:
```

## Step 1: Initialize Simulation State

Reset internal (in-memory) simulation state:

```
sim_portfolio = {
  capital:      $100.00,
  open_trades:  [],
  closed_trades: [],
  daily_pnl:    {},
  regime_history: {}
}

sim_run_header = {
  sid:        [SID from orchestrator],
  run_date:   [today's date],
  sim_start:  [sim_start],
  sim_end:    [sim_end],
  N:          [lookback days],
  universe:   [list of tickers],
  source:     "BACKTEST"
}
```

This state exists only for the duration of this run. It is NOT written to `state/portfolio_ledger.json`.

---

## Step 2: Load Historical OHLC Data

**Tool:** `mcp__robinhood-trading__get_equity_historicals`

Fetch daily OHLC for all universe tickers over `[sim_start – 7 calendar days]` through `[sim_end]`. The extra week provides lookback history for signal computation on Day 1 of the simulation.

Store as:
```
historical_ohlc[ticker][date] = {
  open:   float,
  high:   float,
  low:    float,
  close:  float,
  volume: int
}
```

**Data quality gates (enforce before proceeding to Step 3):**
- All tickers returned data (no missing symbols — abort and report if any fail)
- No gaps > 1 trading day within the simulation range
- Volume > 0 on every day (confirms market was open)
- If a ticker fails the quality gate, remove it from the universe for this run and log the removal

---

## Step 3: Daily Simulation Loop

Iterate over each trading day D from `sim_start` to `sim_end` (skip weekends and market holidays).

---

### 3.1 Generate Signals — Information Cutoff Rule

**This is the most critical discipline in v1.0. State the cutoff explicitly at the top of every simulated day.**

Declare: `"Simulating [DATE]. Using price data ≤ [DATE] only."`

```
Allowed on simulation day D:
  ✅ historical_ohlc[ticker][date ≤ D]   — all price and volume history up to and including D
  ✅ sim_portfolio.regime_history         — regime classifications from prior simulated days
  ✅ Price-derived technical signals (computed from OHLC only — see below)

Not allowed:
  ❌ News articles, earnings results, or analyst commentary published after market close on D
  ❌ Knowledge of price movements after D
  ❌ Any information that would not have been available at market open on D+1
```

**Price-derived signals allowed (computed from historical_ohlc only):**

| Signal | Computation |
|--------|-------------|
| N-day SMA / EMA | Computed from prior closes ≤ D |
| % change | (close[D] − close[D-1]) / close[D-1] |
| Volume ratio | volume[D] / avg(volume[D-20 to D-1]) |
| Distance from 52-week high/low | (close[D] − min/max of prior 252 days) / close[D] |
| Intraday range | (high[D] − low[D]) / close[D] as volatility proxy |
| Momentum | close[D] vs close[D-5], close[D-20] |

---

### 3.2 Detect Market Regime

Using only signals from Step 3.1, classify today's simulated regime:
- **BULL:** Broad index SMA uptrend + positive momentum across sectors
- **BEAR:** Broad index SMA downtrend + negative momentum
- **SIDEWAYS:** Mixed signals, no clear direction

Record result in `sim_portfolio.regime_history[D]`.

Apply regime sizing multipliers (same as live pipeline):
- BULL: cap × 1.5 | SIDEWAYS: cap × 1.0 | BEAR: cap × 0.5

---

### 3.3 Generate Trade Ideas

For each ticker in the universe with a significant price signal on day D:
- State the thesis (direction, catalyst, signal type)
- Note: this step uses the Claude layer and is subject to look-ahead bias — treat output as a plumbing check, not a signal quality test

---

### 3.4 Counter-Thesis Validation

Run `counter_thesis.md` gates **identically** to the live pipeline for each candidate:

```
Step 0b: Price reality check
         entry_assumed vs historical_ohlc[ticker][D].close
         If difference > 10% → REJECT (price error)

Steps A–E: Macro / company / structure / sentiment / timing risks
           (v1.0 caveat: these use Claude's reasoning which has look-ahead contamination)

Step F: Re-check price validity (same as 0b)

Step G: Binary gap protocol
        If earnings or FOMC falls on day D+1 (verifiable from historical record):
          Recalculate R:R assuming 8–12% gap
          If gap-adjusted R:R < 1:2 → WATCH-ONLY

Decision:
  WEAK   → APPROVE (full base size)
  MEDIUM → APPROVE (75% of base size)
  STRONG → REJECT (do not simulate)
```

Record CT rating and flags for each candidate in the trade log.

---

### 3.5 Expected Value Evaluation

Compute running EV from trades closed so far in the simulation:

```
running_win_rate = wins / (wins + losses)   [starts at 0; skip if < 5 closed trades]
avg_win_pct      = mean(win_pct for closed wins)
avg_loss_pct     = mean(loss_pct for closed losses)
EV               = (running_win_rate × avg_win_pct) − ((1 − running_win_rate) × avg_loss_pct)

If EV < 0 (and ≥ 10 closed trades): flag as NEGATIVE_EV — reduce size by 50%
If EV ≥ 0.2: flag as POSITIVE_EV — proceed at normal size
```

---

### 3.6 Sector Exposure Control

Run `sector_exposure_control.md` gates **identically** to the live pipeline:

```
Map ticker → sector using state/sector_exposure.json → _sector_map

Check against internal sim_exposure (NOT the live sector_exposure.json):
  Soft cap: 40% of $100 sim capital per sector
  Hard cap: 50% — never exceeded under any condition
  Max positions per sector: 2

Decision (same logic as live):
  Within limits                                → APPROVE
  Exceeds soft cap + WEAK CT + R:R ≥ 1:3      → ALLOW as Concentrated Bet (size −30–50%)
  Exceeds soft cap + MEDIUM CT                 → REDUCE or DELAY
  Exceeds hard cap (> 50%)                     → REJECT unconditionally
  3rd position in same sector                  → REJECT unconditionally
```

Update `sim_exposure` (internal only — not written to `state/sector_exposure.json`).

---

### 3.7 Trade Execution Simulation

For each approved candidate:

```
Entry price = historical_ohlc[ticker][D+1].open   (next-day open)
Shares      = (allocation_pct × sim_portfolio.capital) / entry_price
Record:
  {ticker, direction, entry, shares, stop, target, ct_rating, entry_date: D+1}
```

Add to `sim_portfolio.open_trades`.

---

### 3.8 Trade Outcome Evaluation

**This is the primary logic being validated in v1.0.**

For each open trade, use `historical_ohlc[ticker][D+1]` to determine outcome:

```
Variables:
  next_open  = historical_ohlc[ticker][D+1].open
  next_high  = historical_ohlc[ticker][D+1].high
  next_low   = historical_ohlc[ticker][D+1].low

For LONG trades:
  stop_hit   = (next_low  ≤ trade.stop)
  target_hit = (next_high ≥ trade.target)

For SHORT trades:
  stop_hit   = (next_high ≥ trade.stop)
  target_hit = (next_low  ≤ trade.target)

Outcome decision table:
  stop_hit=F, target_hit=F → HOLD       (neither triggered; carry position to next day)
  stop_hit=F, target_hit=T → WIN        (close at trade.target)
  stop_hit=T, target_hit=F → LOSS       (close at trade.stop)
  stop_hit=T, target_hit=T → AMBIGUOUS  (both within day's range — sequence unknown)
                              → Conservative rule: record as LOSS (stop assumed first)
                              → Flag: ambiguous=true in trade log
```

**P&L calculation:**
```
LONG  WIN:  (trade.target − next_open) × shares
LONG  LOSS: (trade.stop   − next_open) × shares   [negative]
SHORT WIN:  (next_open − trade.target) × shares
SHORT LOSS: (next_open − trade.stop)   × shares   [negative]
HOLD:       mark-to-market at historical_ohlc[ticker][D].close; re-evaluate next day
```

**Bar resolution note:** Daily OHLC cannot determine intraday stop/target sequence. This is v1.0's known limitation. AMBIGUOUS trades are conservatively recorded as losses. v2.0 will use 5-minute bars to resolve this.

No transaction costs are modeled in v1.0.

---

### 3.9 Update Simulation Portfolio

```
For each closed trade:
  sim_portfolio.capital += realized_pnl
  Move from open_trades to closed_trades
  Record: {ticker, direction, entry, exit, exit_price, pnl, pnl_pct,
           outcome, ambiguous, ct_rating, sector, entry_date, exit_date: D+1}

Record daily snapshot:
  sim_portfolio.daily_pnl[D+1] = {capital, open_count, closed_count, day_pnl}
```

---

### 3.10 Update Signal Performance

For each closed trade, increment running counters by signal type:

```
signal_perf[signal_type] = {
  trades: int,
  wins: int,
  losses: int,
  ambiguous: int,
  win_rate: float,
  avg_win_pct: float,
  avg_loss_pct: float
}
```

## Step 4: End-of-Backtest Metrics

After the loop completes over all N days:

### Performance Metrics
| Metric | Formula |
|--------|---------|
| Total Return % | (sim_portfolio.capital − 100) / 100 × 100 |
| Final Capital | sim_portfolio.capital |
| Total Trades | len(closed_trades) |
| Win Rate | wins / (wins + losses) |
| Avg Win % | mean of win pnl_pct |
| Avg Loss % | mean of loss pnl_pct |
| Expected Value (EV) | win_rate × avg_win − (1 − win_rate) × avg_loss |
| Ambiguous Trade Count | count where ambiguous=true |

### Risk Metrics
| Metric | Formula |
|--------|---------|
| Max Drawdown | max((peak_capital − trough_capital) / peak_capital) |
| Longest Losing Streak | max consecutive losses |
| Risk-Adjusted Return | Total Return / Max Drawdown |

### CT Gate Metrics (key v1.0 output)
| Metric | Value |
|--------|-------|
| Candidates generated | total count |
| Rejected (STRONG CT) | count and % |
| Reduced (MEDIUM CT) | count and % |
| Approved (WEAK CT) | count and % |
| CT false negatives | trades rejected but would have won (review manually) |

### Signal Metrics
- Best performing signal type (highest win rate)
- Worst performing signal type
- Signal accuracy % by regime (BULL / BEAR / SIDEWAYS)
- Sector with highest win rate / highest loss rate

## Step 6: Analysis & Insights

Highlight in the Summary file:

- Which signal types produced positive EV (even with look-ahead caveat, directional insight is useful)
- Which regimes had the best filter performance (CT gate catching more losers in BEAR?)
- Where losses cluster: sector / signal type / regime
- Whether AMBIGUOUS outcomes skew results (high ambiguous count = data quality limitation)
- CT gate false positive rate (STRONG rejections that would have won)

---

## Step 7: Decision Gate

```
If EV > 0.2 AND max_drawdown < 10%:
  → "System plumbing VALIDATED — filters behaving as expected"
  Note: This is NOT a statistical claim. EV is biased by look-ahead.

Else:
  → "System requires review — check CT gate calibration and sector exposure rules"
  Review: Backtest_SignalReport.md for specific failure patterns
```

## Integration with Live Modes

| What BACKTESTING produces | Allowed use in other modes |
|---------------------------|---------------------------|
| Backtest_Summary.md | Human reads to inform manual weight adjustments (RESEARCH / SIMULATION) |
| Backtest_SignalReport.md | Human reads to calibrate CT thresholds or watchlist (manual action only) |
| trade_history.json (tagged BACKTEST) | Reference only — never counted toward EXECUTION gate |
| signal_performance.json (tagged BACKTEST) | Reference only — human decides whether to act on findings |

**Hard rules:**
- Backtest days do NOT count toward the ≥10 simulation days gate for EXECUTION mode
- `state/portfolio_ledger.json` is never modified by BACKTESTING
- `state/regime_state.json` signal weights are never auto-updated by BACKTESTING
- No workflow file is self-modified (Rule 16)

**Human-review pathway after a backtest:**
1. Review Backtest_SignalReport.md → manually edit `state/regime_state.json` weights if warranted
2. Review CT gate analysis → manually adjust `workflows/counter_thesis.md` thresholds if warranted
3. Review sector performance → manually update `resources/watchlist.md` with new tickers
4. All the above are explicit human decisions, not automated changes

## Rules

1. Declare the information cutoff at the start of every simulated day
2. Use only price-derived signals in Step 3.1 — no news, no analyst data
3. Run the CT gate and sector caps identically to the live pipeline
4. AMBIGUOUS outcomes → conservative LOSS (stop assumed first)
5. Never write to `state/portfolio_ledger.json`
6. Always tag trade_history.json and signal_performance.json entries with `source: "BACKTEST"`
7. Do not use backtest results to claim statistical edge without v2.0 rebuild
